Naming conventions

Consistent naming makes code more readable and easier to maintain.

PascalCase: Use for types (classes, interfaces, records, enums), public members (properties, methods, events), namespaces, and file names.

camelCase: Use for local variables and method parameters.

Interfaces: Prefix interface names with I (e.g., IEnumerable, IDisposable).

Constants and readonly fields: Use PascalCase, but avoid SCREAMING_CAPS.

Private fields: Prefix private or internal fields with an underscore (_) followed by camelCase (e.g., _myPrivateField). Static private fields are sometimes prefixed with s_.

Boolean properties/variables: Prefix with a verb to frame them as a question (e.g., isComplete, hasStarted).

Descriptive names: Favor meaningful names over abbreviations (e.g., horizontalAlignment is clearer than alignHoriz).

Code formatting

Formatting rules ensure consistency and readability across a project.

Indentation: Use two spaces for each level of indentation, not tabs.

Statements per line: Write only one statement per line.

Spacing: Add a single space after keywords like if, for, and while, as well as around binary operators like == and +.

Braces: Place opening and closing braces on their own lines (Allman style), and always use braces even for single-line conditional statements.

Blank lines: Use blank lines to separate logical sections of code, such as between method and property definitions.

Documentation and comments

Comments should clarify why a piece of code exists, not just what it does.

XML comments (///): Use for documenting public members like classes, methods, and properties. Visual Studio uses these to provide IntelliSense and generate documentation.

Single-line comments (//): Use for short explanations or notes.

Avoid outdated comments: Do not use C-style multi-line comments (/* ... */) for documenting code. Reserve them for temporarily commenting out code.

Keep comments up-to-date: If you modify code, update its corresponding comments to prevent confusion.

Code organization

Logical organization of code is essential for maintainability.

Single responsibility principle (SRP): Each class should represent one logical object and each method should have one main purpose, making them more focused and easier to manage.

Small methods: Keep methods small and focused on a single task. If a method grows too large, refactor it into smaller, more specific methods.

File-scoped namespaces: Place the namespace declaration at the top of the file to reduce indentation.

One type per file: For clarity and ease of navigation, declare each top-level type (class, interface, etc.) in its own file.

Best practices

These standards improve code quality, performance, and robustness.

Use var wisely: Use implicit typing (var) only when the type is obvious from the right-hand side of the assignment.

Exception handling: Use try/catch/finally blocks to handle errors and release resources. Catch only specific, predictable exceptions.

The using statement: Use the using statement for disposable types to ensure resources are cleaned up properly.

Avoid magic numbers: Replace hard-coded numeric values with named constants or enums to make their purpose clear.

Asynchronous programming: Use the async and await keywords for I/O-bound operations to prevent blocking and improve application responsiveness.

Null-conditional operators: Use the null-conditional operator (?.) to safely access members on potentially null objects, reducing the need for explicit null checks.

Immutable types: Use readonly fields and record types when immutability is desired to prevent accidental changes.

Prefer is and as: Use the is and as operators for safe casting and type checking.
